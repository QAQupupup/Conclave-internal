# 高并发订单系统设计文档

## 1. 背景

某电商平台日均订单量 500 万，峰值 QPS 达到 8 万。现有系统基于单体架构，数据库 MySQL 单机部署，在大促期间频繁出现超卖、数据库连接池耗尽、响应超时等问题。需要设计新的高并发订单系统。

## 2. 核心需求

- 峰值 QPS 8 万，日常 QPS 2 万
- 订单创建到支付完成 P99 延迟 < 500ms
- 库存扣减不可超卖
- 支持分布式部署，水平扩展
- 数据一致性最终一致，允许短暂不一致

## 3. 技术方案

### 3.1 架构选型

推荐微服务架构，订单服务独立部署。流量入口用 Nginx 做负载均衡，后面接多个订单服务实例。订单服务用 Java Spring Boot 实现，数据库用 MySQL 分库分表。

关于缓存方案存在争议：一方认为应该用 Redis 做缓存和分布式锁，另一方认为应该用本地缓存（Caffeine）减少网络开销。

### 3.2 库存扣减方案

库存扣减采用"预扣减 + 异步确认"模式。下单时在 Redis 中预扣库存，支付成功后异步写入 MySQL。

风险点：Redis 宕机可能导致预扣减丢失。需要通过 Redis 集群保证高可用，并在 Redis 不可用时降级到数据库直接扣减。

### 3.3 分库分表策略

按用户 ID 取模分库，每库 16 张表。订单表和订单详情表采用相同分片键，避免跨库 JOIN。

争议点：有观点认为应该按时间维度分表（冷热分离），有观点认为按用户分表更利于查询。

### 3.4 消息队列

使用 RocketMQ 做异步解耦。订单创建后发送消息，库存服务、积分服务、通知服务各自消费。

幂等性保证：消费端通过订单号 + 状态机做幂等校验。

## 4. 部署架构

- 订单服务：8 实例，每实例 4C8G
- MySQL：1 主 2 从，32C128G
- Redis 集群：6 节点，3 主 3 从
- RocketMQ：3 节点

## 5. 监控告警

- Prometheus + Grafana 监控
- 关键指标：QPS、P99 延迟、错误率、库存扣减成功率
- 告警阈值：P99 > 800ms 或错误率 > 1%
