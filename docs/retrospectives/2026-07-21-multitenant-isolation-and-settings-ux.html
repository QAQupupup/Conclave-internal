1. 问题概述

Phase 1b 完成了核心业务表（meetings/messages/events/memories 等）的 tenant_id 列和 DAO 层租户过滤，但在端到端测试中发现以下系统性问题：

  1. 全局资源隔离遗漏：API Keys、Docker Hosts、Net Auth 申请单、Documents 这四类资源没有或缺少完整的 tenant_id 过滤，存在跨租户数据泄露风险。
  2. SQLAlchemy FK 引用 raw SQL 表导致 P0 崩溃：为全局资源表添加 ForeignKey("tenants.id") 后，Base.metadata.create_all() 因 tenants 表由插件 raw SQL 创建（未在 SQLAlchemy metadata 中注册）而抛出 NoReferencedTableError，导致整个测试套件和服务启动失败。
  3. 用户设置 UX 缺口：Settings 页面只有主题/日志开关和一个不可编辑的 API Key 列表，缺少账户管理（修改显示名、修改密码）、团队创建、Key 检测、空状态提示等基础功能。
影响范围：所有多租户部署场景。租户 A 的 API Key 可能被租户 B 使用；Docker 测试全部失败；新用户首次进入设置页面几乎无法完成任何配置。

3. 修复详情

P0 SQLAlchemy ForeignKey 引用 raw SQL 创建的 tenants 表导致 create_all 崩溃 #FK-001
现象
Docker 测试启动时抛出 sqlalchemy.exc.NoReferencedTableError: Foreign key associated with column 'docker_hosts.tenant_id' could not find table 'tenants' with which to generate a foreign key，所有测试失败。
根因
tenants 表由 ensure_tenants_table() 通过 raw SQL CREATE TABLE IF NOT EXISTS 创建，并未注册到 SQLAlchemy Base.metadata。在 ORM 模型中声明 ForeignKey("tenants.id") 后，metadata.create_all() 在排序表依赖关系时找不到被引用表，直接抛出异常。Phase 1b 中 user.py 已经有同样的 FK 声明，但因为测试环境通过 lifespan 先调用插件初始化（raw SQL 建表）再走 SA create_all，所以之前在本地 dev 环境未暴露；Docker 测试中 pytest-xdist 多 worker 并发初始化时竞争条件触发此问题。
修复方式

从所有 ORM 模型（user.py、docker_host.py、document.py）中移除 SA 层的 ForeignKey("tenants.id") 声明，仅保留 mapped_column(Integer, nullable=True, index=True)。外键约束统一由 tenants/service.py::ensure_business_tables_tenant_id() 通过 raw SQL ALTER TABLE ... ADD CONSTRAINT ... REFERENCES tenants(id) ON DELETE SET NULL 添加，这是项目既定模式（参见 ADR-002）。

# 修复前
tenant_id: Mapped[int | None] = mapped_column(
    Integer, ForeignKey("tenants.id", ondelete="SET NULL"), nullable=True, index=True
)
# 修复后
# tenant_id: 由 tenants service 通过 ALTER TABLE 添加外键约束
tenant_id: Mapped[int | None] = mapped_column(Integer, nullable=True, index=True)
影响文件
backend/app/db/models/user.py, docker_host.py, document.py
验证
Docker 测试 544 passed；启动时无 SA 警告。
P1 API Keys 跨租户可见：list/save/delete 缺少 tenant_id 过滤 #ISO-001
现象
切换到不同租户后，API Key 列表会显示其他租户的 Key；删除 Key 时未校验所有权。
根因
services/key_store.py 的 list_keys() 没有按 current_tenant_id() 过滤；save_key() 没有在 INSERT 时填充 tenant_id；delete_key() 没有加 WHERE tenant_id 条件。虽然 _BUSINESS_TABLES 中已包含 api_keys，tenant_id 列已由迁移添加，但 DAO 层完全没有使用它。
修复方式
在 key_store.py 的所有 CRUD 方法中引入租户上下文：list_keys() 查询时加 WHERE tenant_id = :tid OR tenant_id IS NULL（系统 Key 继承），save_key() INSERT 时填充 tenant_id，delete_key() 加双重过滤。
影响文件
backend/app/services/key_store.py
验证
手动切换租户验证 Key 列表隔离；测试通过。
P1 Docker Hosts 全局共享：所有 CRUD 无租户隔离 #ISO-002
现象
Docker Hosts 列表在所有租户间共享；任何租户可删除其他租户添加的 Docker 主机。
根因
docker_hosts 表之前不在 _BUSINESS_TABLES 中，迁移脚本不会为其添加 tenant_id 列；路由层也无任何租户过滤。
修复方式
  1. 在 tenants/service.py::_BUSINESS_TABLES 中加入 "docker_hosts"，由迁移自动添加 tenant_id 列和外键。
  2. 在 routers/docker_hosts.py 中实现 _tenant_filter()，所有 list/get/update/delete 操作均按 (tenant_id = :tid OR tenant_id IS NULL) 过滤，创建时填充 tenant_id。
  3. ORM 模型中添加 tenant_id 列（无 SA 外键，参见 #FK-001）。
影响文件
backend/app/tenants/service.py, routers/docker_hosts.py, db/models/docker_host.py
验证
Docker 测试通过。
P1 Net Auth 申请单无租户过滤：跨租户可见/可操作 #ISO-003
现象
外网访问审批申请单在不同租户间可见；approve/deny 操作无所有权校验。
根因
net_auth.py 使用 raw SQL 操作，之前完全没有 tenant_id 概念。get_pending_for_meeting()、approve_request()、deny_request() 都没有按租户过滤。
修复方式
  1. 在 init_auth_table() 的 ALTER TABLE 中为已存在的表添加 tenant_id INTEGER 列和索引。
  2. create_request() 填充 tenant_id。
  3. get_pending_for_meeting()、list_pending()、get_request() 都加 AND tenant_id = :tid 条件。
  4. approve/deny 通过 get_request() 读取后自然受租户过滤，不额外添加（纵深防御已在 get_request 中）。
  5. expire_pending_requests() 是系统级维护任务，不过滤租户（有意为之，注释已说明）。
影响文件
backend/app/net_auth.py
额外修复
删除了错误创建的 db/models/net_auth.py ORM 模型（该模块全部使用 raw SQL，ORM 模型字段类型不匹配且未被任何代码引用，只会污染 metadata）。

5. 经验沉淀与改进建议

5.1 新增错误模式（应追加到 AGENTS.md §4 和 RETROSPECTIVE_CONVENTIONS §8）

模式：SA 模型与 raw SQL 表混用时 FK 声明不一致

症状：Base.metadata.create_all() 抛出 NoReferencedTableError，指向某个 ForeignKey 找不到被引用表。

根因：被引用表（如 tenants）由插件 raw SQL CREATE TABLE IF NOT EXISTS 创建，未注册到 SQLAlchemy Base.metadata。

规则：对于 raw SQL 创建的表，不要在 SA ORM 模型中声明 ForeignKey；外键约束统一通过 raw SQL ALTER TABLE ... ADD CONSTRAINT 在表创建后添加。SA 模型中仅声明列类型和索引。

模式：误建 ORM 模型污染 metadata

症状：raw SQL 模块已经有自己的 CREATE TABLE，又在 db/models/ 下建了一个 ORM 模型类，导致 create_all 尝试创建同名表，可能产生类型冲突或表已存在错误。

规则：新增 ORM 模型前先确认该模块是否已有 raw SQL DDL。如果有，要么统一迁移到 ORM（一次性），要么完全不要建 ORM 模型。不要混用。

5.2 多租户隔离 Checklist 模板

以后为任何资源表添加多租户支持时，必须逐项确认：

  1. ✅ 表已加入 tenants/service.py::_BUSINESS_TABLES（自动迁移）
  2. ✅ INSERT 时填充 tenant_id = current_tenant_id()
  3. ✅ SELECT 列表加 WHERE tenant_id = :tid OR tenant_id IS NULL（系统资源继承场景）或严格 WHERE tenant_id = :tid
  4. ✅ UPDATE/DELETE 加 WHERE tenant_id 条件，防止跨租户操作
  5. ✅ GET by ID 必须校验 tenant_id，不能仅按 ID 查询
  6. ✅ 不在 SA 模型中对 raw SQL 表声明 ForeignKey
  7. ✅ 测试：创建两个租户 → 各插入数据 → 互相查询不到对方数据

5.3 应补充的测试

  • 多租户隔离集成测试：创建 tenant_a 和 tenant_b，在 tenant_a 下创建 Key/Docker Host/Document/NetAuth request，断言 tenant_b 下查询不到。
  • profile/change-password API 端点单元测试。
  • SA metadata.create_all 冒烟测试：在全新数据库上直接调用 create_all，确保不会因为 FK 引用问题失败。
