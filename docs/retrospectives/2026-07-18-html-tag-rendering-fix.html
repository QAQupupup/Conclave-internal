1. 问题概述

用户反馈会议页面中消息内容的 <p>、<strong> 等 HTML 标签被当作纯文本直接展示（如 <p>各位专家好</p> 原样显示），而非渲染成段落和加粗效果。这是一个影响阅读体验的显示缺陷。

影响范围：Meeting 页消息正文、Report 页富文本块（执行摘要、关键问题、团队配置等）、claim 溯源预览弹窗——所有使用 sanitizeRich 的位置均受影响。

3. 修复详情

P1 已修复 HTML 标签被当作纯文本展示，白名单形同虚设
现象：消息内容中的 <p>、<strong> 等标签在页面上原样显示为文本字符串，没有渲染成段落间距和加粗效果。
根因：sanitizeRich 函数在处理内容时，第一步调用了 formatMessageContent，而该函数内部第 84 行执行了 escHtml(text)——把所有 < 转成了 &lt;。之后将转义后的字符串赋给 tmp.innerHTML 时，浏览器看到的是 &lt;p&gt;，将其当作文本节点而非 HTML 元素。因此 RICH_TAG_WHITELIST（包含 P、STRONG 等）的 has(tag) 判断永远匹配不到真正的元素节点，白名单完全失效。
修复方式：重构 sanitizeRich 的处理顺序： 
// 修复前（错误）：
const formatted = formatMessageContent(html); // escHtml 把 < 转成 <
tmp.innerHTML = formatted; // 浏览器当成文本，白名单失效

// 修复后（正确）：
tmp.innerHTML = html; // 先让 DOM 解析真正的 HTML 元素
// walk 时：白名单元素保留并过滤属性，非白名单元素提取文本
// 对文本节点调用 formatMessageContent 处理 Conclave 语义标记
影响文件：frontend/src/lib/format.ts
验证方式：浏览器 JS 注入 <p>测试<strong>加粗</strong></p>，确认 <p> 和 <strong> 被渲染为真正的 DOM 元素（querySelectorAll('p').length === 1）。

转义顺序错误 sanitize 函数在 DOM 解析前先做了全局 escHtml，导致白名单标签也变成转义文本，白名单机制完全失效 sanitize 应先让 DOM 解析原始 HTML，在元素节点层面做白名单过滤；escHtml 只用于纯文本节点的内容处理

5. 经验沉淀与改进建议

  • sanitize 与 escHtml 的职责必须分离：sanitize 负责元素级白名单过滤，escHtml 负责文本节点级转义，两者不能在 DOM 解析前叠加使用
  • Conclave 语义标记（[fact]、claim-xxx 等）只在文本节点上处理，不干扰 HTML 元素结构
  • 建议增加单元测试覆盖 sanitizeRich：输入含 HTML 标签的字符串，断言输出中白名单标签被保留为真正元素、非白名单标签被移除、Conclave 标记被正确转换
