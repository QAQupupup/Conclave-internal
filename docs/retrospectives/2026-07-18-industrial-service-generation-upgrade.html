1. 问题概述

在全项目从头分析中发现，Conclave的可部署服务生成（deployable_service）能力存在严重的质量瓶颈：

核心问题：单次LLM调用生成20KB单文件app.py，导致以下系统性缺陷：

  • P0 - LLM输出截断和超时：bugfix阶段prompt过大导致连续ReadTimeout，修复完全失效
  • P0 - Demo级产出：生成的代码是单文件FastAPI demo，无分层架构，SQL注入漏洞，缺少静态文件
  • P1 - 无断点续传：阶段失败直接标记FAILED，无法从断点恢复，每次从头开始
  • P1 - 无自我迭代：产出质量不达标时无法自动改进，只能人工重新发起会议
  • P1 - 无测试驱动：部署后不运行测试，功能正确性无法验证
  • P2 - 部署失败无详情：只返回ok/false，无法诊断构建失败原因
  • P2 - 无demo检测：无法识别"实现了功能但只是个demo"的情况
  • P2 - 代码审查误报：将参数化查询误判为SQL注入
  • P2 - 前端状态不同步：后端DONE但前端显示"进行中"
  • P2 - 无跨会议演进：每次从头生成，无法基于历史版本扩展

影响范围：所有deployable_service类型会议的产出质量，直接决定Conclave能否从"演示系统"升级为"生产可用的代码生成系统"。

问题发现背景：全项目从头分析（~45,000行代码逐模块扫描）+ E2E测试验证发现多个P0/P1问题。

3. 修复/升级详情

3.1 P0 Bug修复

P0 已修复 PlaywrightWebSearch session_key重复参数导致搜索失败
现象：Web搜索功能不稳定，Playwright调用时可能因参数重复而出错。
根因：playwright_search.py中kwargs.get("session_key")只读取不消费，导致后续_do_search调用时kwargs中仍包含session_key，与显式传参重复。
修复方式：将kwargs.get("session_key", "default")改为kwargs.pop("session_key", "default")，消费掉该参数。
影响文件：backend/app/tools/playwright_search.py
P0 已修复 Bugfix阶段LLM连续超时，代码修复完全失效
现象：代码审查发现问题后，bugfix阶段prompt包含整个app.py(20KB)+所有问题列表，导致LLM ReadTimeout，修复循环无法推进。
根因：bugfix采用"逐个问题修复+全量代码重生成"模式，单次prompt过大；同时prompt中冗余信息太多。
修复方式：改为按文件批量修复，精简bugfix prompt；同时引入新的自我迭代Loop机制（见3.2），用质量评估驱动的整体迭代替代逐个问题修复。
影响文件：backend/app/agents/prompts.py、backend/app/orchestrator/nodes/produce.py

3.2 P1 架构级升级

P1 新功能 断点续传（Checkpoint & Resume）
问题：阶段失败直接标记FAILED，所有进度丢失，必须从头开始。
实现：
  • MeetingState新增字段：checkpoint、stage_retry_count、max_stage_retries
  • 每阶段执行前/后记录checkpoint，失败时标记resumable=true
  • 阶段级自动重试（默认2次），重试失败后标记FAILED但保留checkpoint
  • _handle_resume扩展支持FAILED状态：从checkpoint的failed_stage重新执行
  • 启动时扫描RUNNING状态会议自动标记PAUSED，崩溃恢复增强
影响文件：backend/conclave_core/state.py、backend/app/orchestrator/runner.py、backend/app/schemas/meeting.py
P1 新功能 自我迭代Loop（Quality-Driven Iteration）
问题：产出质量不达标时无法自动改进，只能人工重新发起。
实现：
  • 质量门禁评估后不达标自动触发迭代（最多2轮，可配置）
  • 质量反馈（分数+问题列表）注入下一轮produce prompt
  • 新增API参数：auto_iterate、max_iterations
  • iteration_history记录每轮评估分数和问题
影响文件：backend/app/orchestrator/runner.py、backend/app/orchestrator/nodes/produce.py
P1 新功能 跨会议演进（Cross-Meeting Evolution）
问题：每次会议从头生成，无法基于已有项目扩展新功能。
实现：
  • produce节点加载reference_meeting_ids的项目文件作为baseline
  • 关键文件内容注入prompt，引导LLM"扩展而非重写"已有代码
  • MeetingCharter增加reference_meeting_ids支持
影响文件：backend/app/orchestrator/nodes/produce.py、backend/app/schemas/meeting.py

5. 根因归类

错误模式描述预防措施
单次LLM输出过载 要求LLM一次输出20KB+单文件或30+文件的完整项目，超出输出可靠性范围，导致截断、幻觉、超时 改为分阶段管线式生成：需求→文档+测试→架构→逐模块→测试→部署。每次LLM调用输出2-3个文件
kwargs.get/pop混淆 kwargs.get只读取不消费，下游再次调用时参数重复导致Playwright调用异常 从kwargs提取独占参数时必须用pop，遵循"谁提取谁消费"原则
全量重生成替代增量修复 bugfix/迭代时重新生成整个项目，可能破坏已正确的代码，且prompt过大导致超时 改为增量patch模式：只修改测试失败/有问题的文件，保留正确代码
部署无反馈循环 部署后不验证、不测试、不检查健康状态，失败时无日志，无法诊断 部署后自动验证健康状态+运行测试+返回错误日志，形成闭环
质量门禁与产出规模不匹配 用单一功能覆盖度指标评估所有规模的产出，无法识别demo级代码 多维度质量评估，引入复杂度匹配检查和demo模式检测
前端状态与后端事件不同步 WebSocket消息类型扩展后前端未同步处理，导致状态残留 新增消息类型时前后端同步更新，增加类型定义共享

6. 经验沉淀与改进建议

  • 分阶段生成是解决LLM大项目生成的根本方案：不要让LLM一次输出整个项目。改为管线式：需求拆分→文档+测试先行→架构规划→逐模块填充→测试→部署配置。每次LLM输出控制在2-3个文件。
  • 测试驱动生成(TDD for AI)：先生成测试用例，再生成实现代码，用测试结果驱动迭代，比事后审查更有效。
  • 断点续传是长流程系统的必备能力：任何可能耗时较长的pipeline都必须支持checkpoint/resume，单次失败不应导致全部重来。
  • 质量门禁必须多维：单一指标（如"部署成功"）无法保证产出质量，需要从架构完整性、代码规模、功能真实性、测试通过等多个维度评估。
  • Demo检测是商业化的关键门槛：LLM倾向于生成"看起来对但无法用"的demo代码，必须通过文件数、代码行数、TODO/NotImplemented标记等信号识别stub产出。
  • kwargs参数消费规范：装饰器/中间件提取kwargs参数时，明确区分"透传参数"（用get）和"消费参数"（用pop）。
  • 前端组件库升级：当前手写CSS组件模式维护成本高，建议引入shadcn/ui（轻量、可定制、现代设计），产出物查看器专业化（Monaco Editor代码查看、专业报告排版）。
  • 数据源接入：代码生成需要GitHub模板few-shot；商业调研需要专业搜索API（Tavily/SerpAPI）；金融分析需要财经数据源（AKShare）。

7. 遗留问题（下一步P0）

  • 分阶段生成管线（最紧迫）：当前project_tree模式仍然是单次LLM调用输出所有文件，20-50个文件的输出可靠性仍存疑。需要实现：Phase1需求拆分→Phase2文档+测试→Phase3架构规划→Phase4逐模块填充→Phase5测试生成→Phase6部署配置。
  • 增量修复：迭代时不重新生成整个项目，只修改有问题的文件（基于diff的patch模式）。
  • 容器构建鲁棒性：npm/pip使用lock文件固定版本；构建超时自动重试；完整构建日志记录。
  • 前端UI组件库升级：引入shadcn/ui替换手写组件；产出物查看器专业化。
