背景与问题清单

在多租户隔离（Phase 2b）完成后，代码库暴露出一批基础架构层面的技术债：部分 ORM 模型字段重复定义、Embedding 同步调用阻塞事件循环、事件总线在多副本部署下会导致 WS 消息丢失、异常/日志/事件 Schema 缺少统一抽象、租户级配置无法覆盖默认值、MeetingState 单模型过重等。这些问题在未来横向扩容、接入真实 LLM/Reranker 时必然踩坑。

P0（必修、多副本下必崩）

  1. ORM 公共字段重复：id/created_at/updated_at/tenant_id 在每个模型里手写，容易漏加 index=True 或漏掉多租户过滤。
  2. Redis Pub/Sub 事件总线桥接缺失：单实例内存总线在多副本部署时，WS 订阅者连在 A 副本、事件发布在 B 副本，消息必然丢失。
  3. Embedding 客户端同步 + Reranker 无抽象：httpx.post 同步调用阻塞 asyncio 事件循环；Reranker 直接写死在 SiliconFlow，无 Protocol 抽象无法切换。

P1（能力缺失，影响扩展）

  1. 异常体系不统一：业务代码抛 HTTPException、ValueError、RuntimeError 混用，错误码散落在路由里，前端无法稳定解析。
  2. 租户级配置未启用：tenants.settings 列为 JSONB 但从未读取，租户无法自定义 LLM/Embedding/Reranker 提供商。
  3. asyncio 事件循环单例泄漏：Redis 客户端、事件总线监听任务等模块级单例未做循环感知，pytest-xdist 下出现大量 "attached to a different loop"。

P2（代码质量与长期维护）

  1. 日志门面不统一：logger/log_bus/audit 三套调用方式并存，contextvars（request_id/meeting_id）注入不彻底。
  2. 事件 payload 裸 dict：事件载荷是无类型 dict，无法做 schema 版本化与自动校验。
  3. MeetingState 单模型过重：一个 Pydantic 模型承载 core/debate/borrow/iteration/observability 五大块字段，演进困难。

P3（加固项）

  1. WS 入站消息无校验：客户端发送的消息直接 json.loads 使用，畸形消息会 500。

踩坑记录：asyncio 事件循环单例泄漏

症状 Docker 容器内 pytest 544 个测试全部跑通后 teardown 阶段大量报错： 
Task <Task pending coro=<InMemoryEventBus.publish() running at events.py:118>>
  got Future <Future pending> attached to a different loop
RuntimeError: Event loop is closed
sqlalchemy.pool.impl.AsyncAdaptedQueuePool: Exception closing connection
错误出现在 ERROR at teardown，导致 pytest 退出码 1。

根因

AGENTS.md §4.1 明确规定"模块级禁止直接实例化 asyncio 原语"，但 app/db/redis.py 的 _redis 全局客户端和 app/events.py 中新加的 Redis Pub/Sub 监听任务都没有做循环感知：

  1. 每个 pytest-xdist worker 进程内，pytest-asyncio 为每个 async test 创建新的 event loop。
  2. 第一次跑带 TestClient 的测试时，lifespan 调用 init_redis()/bus.start()，在 loop-A 上创建 Redis 连接池和 pubsub listener task。
  3. 测试结束，loop-A 关闭；但全局 _redis、bus._redis_pubsub、bus._redis_listener_task 仍然持有旧引用。
  4. 下一个测试在 loop-B 上调用 bus.publish()，await redis_client.publish() 把协程提交给 loop-B，但连接池的 Future 绑定在 loop-A → "attached to a different loop"。
  5. Teardown 时 close_redis()/dispose_async_engine() 尝试在已关闭的循环上 close 连接 → "Event loop is closed" 连锁。

修复方案（循环感知模式）

完全对齐 AGENTS.md §4.1 和 app/db/engine.py 已有的 _ensure_engine 模式：

  • app/db/redis.py 新增 _redis_loop 追踪，_safe_get_client() 在调用时检查 current_loop is _redis_loop and not _redis_loop.is_closed()，不匹配则丢弃旧引用、返回 None 让调用方降级。
  • app/events.py 在 InMemoryEventBus 上新增 _loop 字段，_redis_available() 方法做循环检查；start() 检测循环切换时先丢弃旧状态再在新循环上启动；stop() 仅在绑定循环上执行 await 清理，否则只丢引用；_redis_publish() 循环不匹配时直接 return（静默降级）。
  • _redis_listener() 主循环主动检测 loop.is_closed()，并捕获 RuntimeError("Event loop is closed") 主动 break，不在已关闭循环上 await asyncio.sleep(1)。
  • 关键纪律（AGENTS.md §4.1 规则 3）：循环不匹配时不跨循环调用 await xxx.close()，只清除引用让 GC 回收——跨循环 close 本身就会抛 "attached to a different loop"。
修复后效果 Docker 容器内 pytest 全量跑：544 passed, 11 skipped, 0 failed，退出码 0，不再有 teardown ERROR。

经验沉淀

  1. 模块级 asyncio 原语 = 定时炸弹：任何模块级持有 asyncio.Lock/Semaphore/Task/aioredis.ConnectionPool 等的单例，必须循环感知。pytest-xdist 多 worker 下事件循环频繁重建，是最容易暴露这类问题的场景。
  2. 跨循环释放资源是反模式：AGENTS.md §4.1 规则 3 非常重要——在 sync 代码或不同循环上 await engine.dispose()/await redis.close() 必然抛错。正确做法是"丢引用让 GC 回收"，或另起线程 + 新 asyncio.run()（engine.py 已有范例）。
  3. Redis Pub/Sub 桥接必须有 instance_id 回环防护：否则本地 publish → Redis → 本地 listener → 再 publish → 死循环。
  4. Pydantic 化 event payload 的收益大于迁移成本：schema_version 字段为未来不兼容变更预留了通道，WS 前端可以按版本做适配。
  5. 租户级配置必须有白名单：不能让租户随便覆盖任意 settings 字段，否则会成为注入/配置漂移入口。白名单 key 写死在 settings_override.py 里。
