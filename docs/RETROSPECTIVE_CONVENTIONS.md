# Conclave 修复报告归档规范

> **版本**: 1.0
> **生效日期**: 2026-07-16
> **适用范围**: 所有代码修复、Bug修复、架构调整后的总结报告

---

## 1. 为什么需要修复报告

每次系统性修复（无论是Bug修复、架构重构还是功能迭代），都会产生大量代码变更和决策。修复报告的作用：

1. **经验沉淀**：记录Bug根因，避免重犯同类错误
2. **增量追溯**：后续AI/开发者可以基于历史报告快速理解上下文
3. **审计合规**：每次重大变更都有完整记录
4. **知识传递**：新成员可以通过报告快速了解系统演进历史

---

## 2. 报告存放位置

所有修复报告统一存放在：

```
docs/retrospectives/
├── YYYY-MM-DD-{slug}.html    # HTML格式的完整报告（主要格式）
└── YYYY-MM-DD-{slug}.md      # Markdown格式（可选，简要版本）
```

- **目录**：`docs/retrospectives/`（如不存在需创建）
- **命名规则**：`YYYY-MM-DD-{简短描述}.html`，例如：
  - `2026-07-16-docker-startup-and-ux-fixes.html`
  - `2026-07-14-architecture-refactor.html`
- **格式**：优先使用 HTML 报告（通过 html-report skill 生成），视觉效果好、可直接在浏览器打开

历史报告（之前存放在 `docs/audits/` 下）保持原位不动，新报告统一放 `docs/retrospectives/`。

---

## 3. 何时生成报告

以下情况**必须**生成修复报告：

| 场景 | 报告类型 | 示例 |
|------|---------|------|
| 修复阻断性Bug（P0/P1） | 完整HTML报告 | Docker启动失败、服务崩溃 |
| 一轮系统性修复（≥5个Bug） | 完整HTML报告 | 深夜Bug修复Session |
| 架构级重构 | 完整HTML报告 | DB层统一、AgentRuntime重构 |
| 安全修复 | 完整HTML报告 | Docker socket加固 |
| 单个小Bug修复（<3个文件） | 可不生成，但需在commit message中详细描述 | 拼写错误、样式微调 |
| UI/UX调整 | 视影响范围决定 | 样式重构需要，颜色微调不需要 |

## 4. 报告必须包含的内容

每份修复报告**必须**包含以下章节：

### 4.1 Commit区间

```markdown
## Commit区间
- 起始Commit: <commit-hash> (<commit-message>)
- 结束Commit: <commit-hash> (<commit-message>)
- 本次修复的commit列表:
  - <hash> <message>
  - <hash> <message>
```

通过 `git log --oneline <start>..HEAD` 获取。起始commit是本轮修复开始前的最后一个commit，结束commit是本轮修复完成并提交后的commit。

### 4.2 问题概述

- 发现了多少个问题
- 问题的严重程度分布（P0/P1/P2/P3数量）
- 影响范围（哪些模块/功能受影响）

### 4.3 修复详情

每个Bug必须包含：
- **现象**：用户看到什么/发生了什么
- **根因**：为什么会发生（技术层面，不是表面原因）
- **修复方式**：怎么修的
- **影响文件**：哪些文件被修改
- **验证方式**：怎么确认修好了

### 4.4 根因归类

将Bug按错误模式归类（参考历史报告中的8类典型错误模式），帮助识别系统性问题。

### 4.5 经验沉淀

从这次修复中学到了什么？如何预防同类问题？列出具体的改进建议（加测试、加门禁、改流程等）。

### 4.6 遗留问题

未在本轮修复的问题，记录下来供后续迭代。

### 4.7 验证结果

- 编译检查结果
- 测试结果
- Docker构建和运行状态

---

## 5. Commit 消息规范

每次修复提交时，commit message 应附带报告引用：

```
<type>: <description>

Refs: docs/retrospectives/2026-07-16-xxx.html
```

对于包含多个commit的一轮修复，最后一个commit（合并性commit）必须引用报告路径。

---

## 6. AI 辅助修复时的工作流

当AI（如Trae）执行修复任务时，遵循以下流程：

1. **开始修复前**：
   - 阅读 `docs/retrospectives/` 下最近的报告，了解近期修复历史
   - 阅读 `PROJECT_CONVENTIONS.md` 了解项目规范
   - 确定本轮修复的起始commit（`git rev-parse HEAD`）

2. **修复过程中**：
   - 每完成一批相关修复，运行编译检查（py_compile/tsc）
   - 记录每个Bug的现象、根因、修复方式（不要靠记忆，边修边记）

3. **修复完成后**：
   - 运行完整验证（语法检查、构建、Docker启动）
   - 生成修复报告（使用 html-report skill）
   - 报告中明确标注commit区间
   - 将报告保存到 `docs/retrospectives/`
   - 提交代码，commit message引用报告路径

## 7. 典型错误模式参考（持续更新）

以下是历史修复中归纳的典型错误模式，新报告中如发现新模式应追加到本节：

| 模式 | 描述 | 预防措施 |
|------|------|---------|
| API风格混用 | SQLite语法套用到PostgreSQL（conn.execute vs cursor） | DB驱动迁移建立检查清单 |
| 字符串校验代替语义校验 | startswith("{")判断代码有效性 | 用AST解析、schema校验 |
| 重命名后遗留不一致 | 部分引用未更新、存量数据未兼容 | 全局搜索+normalize兼容层 |
| 事件时序错误 | 先通知后决策导致UI闪烁 | "先决策后通知"原则 |
| API契约不一致 | 同端点不同分支返回不同结构 | TypeScript/Pydantic双重约束 |
| 异常路径无收敛 | 异常后状态僵死、无事件通知 | try/except/finally确保状态收敛 |
| BOM污染 | Windows编辑器添加BOM导致解析失败 | .editorconfig + CI BOM检测 |
| 修复引入回归 | 修改代码后不运行编译检查 | CI加py_compile/tsc门禁 |
| 导航路径断裂 | 操作后无明确出口/入口 | 每次操作都有明确的返回路径 |
| 重复入口不一致 | 同一功能多个入口，交互不同 | 统一交互模式，或明确差异化定位 |

## 8. 报告模板

生成新报告时，可参考 `docs/retrospectives/` 下最新的报告作为模板，保持格式一致。快速骨架：

```html
<!-- Generated by Trae Work -->
<!DOCTYPE html>
<html lang="zh-CN">
<head>...（样式，复用历史报告的CSS）...</head>
<body>
<div class="container">
  <header class="report-header">
    <h1>修复报告标题</h1>
    <p class="subtitle">一句话概述</p>
    <div class="report-meta">
      <span>📅 日期</span>
      <span>🔧 起止Commit: <code>abc1234</code> → <code>def5678</code></span>
      <span>📝 N个修复项</span>
    </div>
  </header>
  <!-- 统计卡片 -->
  <!-- 章节内容 -->
</div>
</body>
</html>
```
