# ADR-003: 插件三层分级（CORE / CROSSCUTTING / OPTIONAL）

| 字段 | 值 |
|------|-----|
| 编号 | ADR-003 |
| 状态 | Accepted |
| 日期 | 2026-07-19 |
| 影响范围 | PluginRegistry 加载逻辑、服务启动流程、错误处理与降级策略、监控告警规则、插件准入审核标准 |

## 背景

在 ADR-001 确定插件化架构后，v0.3 内测阶段出现了一个严重的可用性问题：测试环境中 team 插件配置的独立数据库（用于团队维度的报表聚合）由于连接串配置错误，在插件 `async init()` 阶段抛出了 `ConnectionRefusedError`。PluginRegistry 当时的实现是"任何插件初始化失败则整个服务启动失败"，导致核心会议服务完全不可用——而实际上，没有 team 插件，用户依然可以创建个人会议、调用 LLM、生成逐字稿。这是典型的"非核心组件故障拖垮核心服务"的反模式。

类似的场景还包括：

- billing 插件依赖的 Stripe API 临时不可达，但用户创建会议不应该因此被阻断（至少应该支持在配额内继续使用并事后补记账）；
- webhook 插件依赖的消息队列短暂抖动，不应该影响已经在进行中的会议；
- audit 插件写入 ES 集群超时，不应该阻塞业务主链路。

我们需要对插件进行关键性分级，不同层级的插件在初始化失败、运行时异常、依赖不可用时采取不同的容错策略。如果所有插件一视同仁地采用"失败即阻断"策略，系统脆弱性将不可接受；反之，如果全部采用"失败即跳过"策略，auth 插件挂掉时服务会裸奔启动，同样是灾难。

## 决策

插件按关键性分为三层，每个插件必须在入口类中显式声明 `tier: PluginTier` 枚举值：

```python
from enum import Enum

class PluginTier(str, Enum):
    CORE = "core"
    CROSSCUTTING = "crosscutting"
    OPTIONAL = "optional"
```

### 第一层：CORE（核心级插件）

**定位**：没有它，Conclave 核心功能无法安全或正确运行的插件。CORE 插件是核心的延伸，与核心同生共死。

**典型插件**：`auth`（认证鉴权）、`ratelimit`（全局限流，防止 LLM 账单爆炸）、`tenant_resolver`（多租户路由）。

**失败策略**：
- 初始化失败：**阻断服务启动**，进程以非零退出码退出，同时输出 CRITICAL 级别日志和明确的错误码（如 `PLUGIN_CORE_INIT_FAILED:auth`）；
- 运行时钩子抛异常：**中断当前请求**，返回 500，触发告警；不做静默降级（例如 auth 插件异常时绝不允许"默认放行"）；
- 健康检查：CORE 插件不健康时，整体服务 `/healthz` 返回 503，K8s 会重启 Pod。

### 第二层：CROSSCUTTING（横切关注点插件）

**定位**：对业务链路有影响但不构成"功能不可用"的横切能力。失败需要降级但必须强烈告警，运营需要介入。

**典型插件**：`billing`（用量计费）、`audit`（审计日志）、`metrics`（业务指标采集）。

**失败策略**：
- 初始化失败：**服务可以启动**，插件被标记为 `DEGRADED` 状态；启动日志输出 ERROR 级别告警，触发即时告警（PagerDuty/值班群）；
- 运行时钩子抛异常：**捕获异常并降级**，例如 billing 插件失败时，请求继续进行但该次用量记入本地"待补扣队列"（outbox 表），后台重试；audit 失败时先写本地 WAL 文件，ES 恢复后补传；
- 降级持续时间：降级超过 30 分钟自动升级为 P1 告警；超过 24 小时仍未恢复则将该插件状态置为 FAILED，并通过管理事件通知到所有平台管理员；
- 健康检查：`/healthz` 返回 200（服务可用），`/healthz?deep` 返回 degraded 并列出降级的横切插件。

### 第三层：OPTIONAL（可选业务插件）

**定位**：增强型业务功能，缺失时核心体验基本完整，相关功能入口自动隐藏或返回友好提示。

**典型插件**：`team`（团队管理）、`slack`/`feishu`/`zoom`（第三方集成）、`analytics`（高级分析报表）、`template`（自定义模板库）。

**失败策略**：
- 初始化失败：**插件被自动禁用**，注册流程跳过该插件；日志输出 WARN；不触发 PagerDuty，仅在管理后台"插件状态"页显示；
- 运行时钩子抛异常：**捕获异常并禁用该插件对应功能**，例如 slack 插件推送消息失败时，该会议的 Slack 同步按钮变灰并提示"Slack 集成暂不可用"，但会议本身继续正常进行；
- 断路器机制：同一个 OPTIONAL 插件 5 分钟内连续异常超过 20 次，自动触发熔断（Circuit Breaker），10 分钟内不再调用该插件任何钩子，冷却后尝试半开恢复；
- 健康检查：`/healthz` 完全不纳入考量，`/plugins/status` 接口单独展示 OPTIONAL 插件健康状态。

### 加载顺序与依赖

PluginRegistry 按 CORE → CROSSCUTTING → OPTIONAL 顺序加载插件。同一层级内按 `plugin.yaml` 声明的 `dependencies` 做拓扑排序。高层级插件可以依赖低层级插件（如 team 依赖 auth），反之不允许（auth 绝不能依赖 team）。Registry 在启动时会校验依赖方向，违反时直接报错。

## 选项对比

| 选项 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 选项1：不分层，所有插件统一按"失败即阻断"处理 | 实现最简单；状态模型单一，无需区分 tier | 非核心插件故障会拖垮核心服务（已在 v0.3 内测中真实发生）；运维和客户体验极差；任何第三方集成抖动都会变成全局故障 | 否决，已证明不可接受 |
| 选项2：不分层，所有插件统一按"失败即跳过"处理 | 容错性最强，任何插件坏了都不影响核心 | 关键插件（如 auth、ratelimit）故障时服务会裸奔启动，存在严重安全和资金风险；billing 挂了可能跑掉大量未计费用量，造成直接经济损失 | 否决，安全性无法保障 |
| 选项3：两层分级（REQUIRED / OPTIONAL） | 比不分层好，能区分关键与非关键 | 粒度不够，billing/audit 这类"失败不应阻断但必须告警且需降级重试"的插件找不到合适位置——放 REQUIRED 太严，放 OPTIONAL 太松；没有"降级"这一中间态，运维感知不足 | 否决，缺少横切关注点这一关键中间层 |
| 选项4：三层分级 CORE / CROSSCUTTING / OPTIONAL（选定） | 清晰区分"不能挂"、"挂了要降级告警"、"挂了不影响核心"三类场景；策略与业务语义匹配；运维可根据 tier 设置不同告警级别和响应 SLA | 需要 PluginRegistry 实现三套加载/容错逻辑；插件开发者需要准确判断自己插件的 tier（放错 tier 会有风险）；需要配套监控和管理页面展示各 tier 状态 | 选定，分级粒度与 Conclave 的业务场景高度吻合，容错策略可操作性强 |

## 后果

### 正面影响

1. **核心可用性提升**：非核心插件故障不再影响会议创建/LLM 调用等主链路，预计核心服务可用性从 99.5% 提升到 99.9%+；
2. **告警分级明确**：CORE 插件故障触发 P0 立即响应，CROSSCUTTING 触发 P1 半小时响应，OPTIONAL 仅触发 P3 工单，值班团队不再被低优先级告警轰炸；
3. **私有化部署友好**：私有化客户如果没有部署 Slack/Zoom 等集成，只需将这些插件配置为不启用，不会影响启动；
4. **渐进式降级**：CROSSCUTTING 层的 outbox/WAL 降级模式保证即使依赖故障，数据也不会丢失，恢复后自动对账；
5. **插件审核标准清晰**：新插件 RFC 评审时，第一个问题就是"这个插件属于哪个 tier"，答案直接决定它的容错策略、测试标准、上线要求。

### 负面影响

1. **实现复杂度**：PluginRegistry 需要维护三种状态（LOADED / DEGRADED / FAILED）× 三个 tier 的行为矩阵，代码量增加约 300 行；
2. **tier 判定争议**：某些插件（如 `tenant_resolver`）到底算 CORE 还是 CROSSCUTTING 可能存在争议，需要架构组在 RFC 阶段拍板；
3. **降级测试成本**：CROSSCUTTING 插件的降级路径需要专门的故障注入测试（如 Chaos Monkey），CI 流水线需要增加相关用例；
4. **状态可见性要求**：需要提供 `/plugins/status` 管理 API 和前端页面展示各插件状态，否则运维人员不知道哪些插件正在降级运行；
5. **依赖方向约束**：CORE 插件不能依赖 OPTIONAL 插件这条规则需要静态检查工具（自定义 lint 规则）辅助，否则容易在代码评审中被绕过。

### 缓解措施

- 提供 `@plugin(tier=...)` 装饰器，在类声明处强制要求填写 tier，缺少 tier 字段的插件无法通过 Registry 校验；
- 在 `conclave test` 命令中内置 `--chaos` 模式，可随机注入 CROSSCUTTING/OPTIONAL 插件故障，验证降级逻辑；
- 编写《插件 Tier 判定指南》附在设计文档中，给出 10+ 个典型插件的 tier 归属示例和判定 checklist（"故障是否会导致未授权访问？""故障是否会导致直接经济损失？""故障后用户核心会议流程是否可用？"）；
- 在管理后台首页设置"插件健康面板"，用红/黄/绿三色展示各 tier 插件状态，降级/熔断状态醒目提示；
- 编写 ESLint/Pylint 自定义规则 `no-core-import-optional`，禁止 CORE 插件直接 import OPTIONAL 插件的模块，CI 中强制检查。

### Tier 归属初始清单

| 插件 | Tier | 理由 |
|------|------|------|
| auth | CORE | 无认证则所有接口裸奔 |
| ratelimit | CORE | 无限流可能导致 LLM 账单爆炸 |
| tenant_resolver | CORE | 多租户路由错误会导致数据串租户 |
| billing | CROSSCUTTING | 计费失败不能阻断服务但必须补扣 |
| audit | CROSSCUTTING | 审计日志丢失有合规风险，需 WAL 补传 |
| metrics | CROSSCUTTING | 指标丢失影响监控但不影响业务 |
| webhook | CROSSCUTTING | 通知失败不应阻塞会议主链路 |
| team | OPTIONAL | 个人模式可独立运行 |
| slack/feishu/zoom | OPTIONAL | 第三方集成纯增强功能 |
| analytics | OPTIONAL | 高级报表不影响核心会议 |
| template | OPTIONAL | 无自定义模板可用默认模板 |
