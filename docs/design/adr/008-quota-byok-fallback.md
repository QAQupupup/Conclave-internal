# ADR-008: 配额耗尽自动降级（BYOK Fallback）

| 字段 | 值 |
|------|-----|
| 编号 | ADR-008 |
| 状态 | Accepted |
| 日期 | 2026-07-19 |
| 影响范围 | LLM 调用错误处理钩子、Token 选择器 UI 组件、会议运行时状态机、用户个人 API Key 管理、WebSocket 实时通知 |

## 背景

Conclave 会议中大量依赖 LLM 调用：实时语音转写后的 AI 处理、实时摘要更新、行动项提取、会后纪要生成、代码解释等。一次 1 小时的会议平均消耗 30-80 万 Token（取决于发言密度和 AI 功能开启数量）。

根据 ADR-007 的父池切分模型，团队的配额可能在会议进行中耗尽，触发场景包括：

1. **月度预算用完**：团队本月 Token 消耗达到 `monthly_token_budget`，且父链上没有可用余额；
2. **LLM 提供商侧余额耗尽**：API Key 对应的 OpenAI/Anthropic 账户余额用完或信用卡过期，返回 402/429（billing-related）错误；
3. **速率限制触发**：短时间内请求量超过 API Key 的 RPM/TPM 限制；
4. **并发会议超量**：多个会议同时运行，总消耗速度超过预算预警阈值。

配额耗尽发生在会议中途时，如果直接中断 AI 功能，会严重影响用户体验——正在进行的会议突然失去实时摘要、行动项不再被记录、会后无法生成纪要。内测期间有用户反馈："开了 40 分钟的会，最后 20 分钟 AI 突然不工作了，前面的讨论也没记录全。"

我们需要决定配额耗尽时的系统行为，在"控制成本"和"保障会议连续性"之间找到平衡。

关键约束：
- Conclave 不处理充值（见 ADR-007），无法在耗尽时自动扣款续额；
- 用户可能在个人设置中配置了自己的 API Key（BYOK = Bring Your Own Key）；
- 会议场景下不能让用户被技术错误打断，降级必须是平滑的；
- 必须让用户清楚知道当前使用的是哪个配额池、余额状态如何，避免"为什么我个人 Key 被扣费了"的纠纷。

## 决策

采用**软降级策略（Soft Degradation with BYOK Fallback）**：配额耗尽时优先自动切换到用户个人的 BYOK，并通过 UI 明确告知用户；如果用户没有 BYOK，则暂停 AI 功能并提示配置 Key 或联系管理员。

### 降级优先级链

当 LLM 调用因配额/计费问题失败时，按以下顺序尝试降级：

```
1. 团队池（团队继承链上解析到的 API Key，见 ADR-007）
   └─ 失败原因：配额耗尽 / 提供商余额不足 / 速率限制
      ↓
2. 会议发起人的个人 BYOK（如果已配置）
   └─ 失败原因：个人 Key 也不可用 / 未配置
      ↓
3. 会议中其他在线参与者的个人 BYOK（按角色优先级：主持人 > 管理员 > 普通成员）
   └─ 全部失败或无人配置
      ↓
4. 暂停 AI 功能，提示用户操作
```

每一次降级都必须通过 UI 明确告知当前状态，不做静默切换。

### 个人 BYOK 数据模型

用户可以在个人设置中配置自己的 API Key，作为团队配额耗尽时的兜底：

```sql
-- users 表扩展
ALTER TABLE users ADD COLUMN personal_llm_api_key_encrypted BYTEA;
ALTER TABLE users ADD COLUMN personal_llm_provider VARCHAR(50);
ALTER TABLE users ADD COLUMN personal_llm_base_url TEXT;
ALTER TABLE users ADD COLUMN personal_llm_model VARCHAR(100);
-- 用户是否允许自己的 Key 被自己发起的会议使用（默认 true）
ALTER TABLE users ADD COLUMN byok_auto_fallback BOOLEAN NOT NULL DEFAULT TRUE;
-- 用户是否允许自己的 Key 被他人发起的会议兜底（默认 false，需要显式开启）
ALTER TABLE users ADD COLUMN byok_allow_others_meetings BOOLEAN NOT NULL DEFAULT FALSE;
```

个人 BYOK 的使用有明确的授权边界：
- 默认仅在**自己发起/主持的会议**中自动降级使用；
- 其他用户的会议如果要使用你的 BYOK，需要显式开启 `byok_allow_others_meetings`，且仅在会议主持人和其他管理员均无可用 Key 时才会使用；
- 个人 BYOK 的 Token 消耗记录在用户个人名下（`user_usage` 表），用户可在个人设置中查看累计消耗；
- 个人 Key 的费用直接由用户的 LLM 提供商账户承担，Conclave 不收取额外费用。

### 错误检测与降级触发

LLM 调用钩子（`before_llm_call` 和 `on_llm_error`）中实现配额检测和降级逻辑：

1. **预检（before_llm_call）**：在发起 LLM 调用前检查团队有效余额（结合缓存），如果预计本次调用后将超过预算的 95%，在 UI 上显示预警横幅"团队配额即将用完，剩余约 X Token"；
2. **错误识别（on_llm_error）**：识别以下错误类型作为可降级错误：
   - HTTP 402 Payment Required
   - HTTP 429 且错误体包含 `insufficient_quota` / `billing_hard_limit` / `rate_limit_exceeded`（RPM 类限频错误先退避重试 3 次再降级）
   - 服务端配额检查返回的 `QUOTA_EXCEEDED` 业务错误码
3. **降级执行**：按优先级链切换 API Key 并重试当前调用，重试最多 2 次（切换到不同 Key）；
4. **降级成功**：在会议状态中记录当前使用的 Key 来源（`key_source: 'team' | 'personal' | 'member_<user_id>'`），后续同会议内的 LLM 调用直接使用降级后的 Key，避免每次调用都重新尝试失败的 Key；
5. **降级失败**：暂停会议的 AI 功能（但不中断语音通话和转文字），向所有参会者显示提示条。

### 会议运行时状态机扩展

会议（Meeting）的 AI 状态增加以下子状态：

```
AI_ACTIVE（正常使用团队池）
  → AI_DEGRADED_PERSONAL（已降级到发起人 BYOK）
  → AI_DEGRADED_MEMBER（已降级到参会者 BYOK）
  → AI_PAUSED（AI 暂停，等待用户操作）
  → AI_RESUMING（用户配置了新 Key，正在恢复）
```

状态转换通过 WebSocket 实时广播给所有参会者，UI 同步更新状态指示器。

### Token 选择器 UI

会议发起前的 Token 选择器（会议设置面板中的 AI 配置区域）必须清晰显示：

1. **当前使用的配额池**：团队名称（如"Acme Corp > 产品部"）+ Key 来源标签（"团队配额"）；
2. **余额状态**：进度条显示本月已用/总预算，颜色区分（绿色 > 20%、黄色 5%-20%、红色 < 5%）；
3. **预计消耗**：根据会议预计时长和开启的 AI 功能，预估本次会议消耗的 Token 量；
4. **降级预案提示**：如果用户配置了个人 BYOK，显示"配额耗尽时将自动使用你的个人 Key"；如果没有配置，显示"配额耗尽时 AI 功能将暂停"并提供配置入口；
5. **强制选择**：用户可以在发起会议时主动选择使用个人 BYOK（而非团队池），适用于私人会议或测试场景。

### 降级时的 UI 提示

降级发生时，UI 需要给出明确但不打断会议流程的提示：

1. **降级到个人 BYOK**：顶部出现黄色信息条（非模态，不阻挡操作）："团队配额已用完，已自动切换到你的个人 API Key。本次会议后续 AI 消耗将从你的个人账户扣费。[了解详情] [切回团队池（如有新配额）]"；
2. **降级到他人 BYOK**：主持人看到"团队配额已用完，已切换到 [成员名] 的个人 Key"，其他参会者看到"团队配额已用完，AI 功能继续使用备用 Key 运行"；
3. **AI 暂停**：顶部出现红色警告条："AI 功能已暂停——团队配额已用完且无可用的个人 Key。[配置我的 API Key] [联系管理员充值]"，同时提供一个输入框让用户现场粘贴 API Key 后立即恢复；
4. **恢复**：当用户配置了新 Key 或管理员充值后，AI 功能自动恢复（状态从 AI_PAUSED → AI_ACTIVE/AI_DEGRADED），显示"AI 功能已恢复"。

### 速率限制的特殊处理

对于 API Key 的 RPM（Requests Per Minute）/TPM（Tokens Per Minute）速率限制（非余额耗尽），处理策略不同：

1. 首先在 Key 内部做指数退避重试（1s → 2s → 4s，最多 3 次）；
2. 如果仍然 429，且团队内有其他可用 Key（如子团队配置了独立 Key），切换到其他 Key；
3. 不立即降级到个人 BYOK——因为速率限制是暂时的（通常 60s 后恢复），降级到个人 Key 反而可能导致不必要的个人扣费；
4. 持续 2 分钟以上的速率限制才考虑降级，并在提示中说明"当前 Key 触发频率限制"而非"配额耗尽"。

## 选项对比

| 选项 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 选项 A：硬阻断（Hard Stop） | 实现最简单；成本控制绝对可靠（不会产生意外费用）；无需用户授权个人 Key | 会议中途 AI 突然停止，严重影响用户体验和数据完整性（会议记录中断）；用户可能因为一次突发的配额耗尽丢失关键会议内容；反复出现会导致用户对产品失去信任；内测反馈中最差评的方案 | 否决，会议场景对连续性要求极高，硬阻断不可接受 |
| 选项 B：自动降级 BYOK + 明确提示（选定） | 会议 AI 功能最大程度保持连续；用户对费用来源知情（UI 明确提示）；个人 BYOK 作为安全网不强制配置（无 BYOK 用户仍可暂停后手动配置）；降级链路有优先级，避免随意使用他人 Key；符合 SaaS 产品"优雅降级"的最佳实践 | 实现复杂度较高（需要 Key 切换逻辑、状态机扩展、UI 状态同步）；有极小概率产生用户预期之外的费用（但通过明确提示和默认关闭"为他人兜底"来规避）；需要处理部分成功/部分失败的边界情况（如降级后调用到一半 Key 又失效） | 选定，在保障会议连续性的同时通过 UI 透明度和授权边界控制费用风险 |
| 选项 C：管理员可配置策略（Configurable Policy） | 最灵活——管理员可以选择硬阻断/降级/仅预警等策略；满足不同组织的合规需求（如某些企业禁止使用个人 Key 处理工作数据） | MVP 阶段过度复杂：需要策略引擎、策略继承（子团队继承父团队策略）、策略 UI 配置界面、审计日志；不同策略组合的测试矩阵爆炸；大部分中小团队不需要这种灵活性，增加配置负担；延迟核心功能交付 | 否决，MVP 阶段先实现选项 B 的固定策略，v0.5 再根据客户反馈抽象策略配置。管理员可通过禁用"允许个人 Key"组织级设置来达到类似硬阻断的效果 |

## 后果

### 正面影响

1. **会议连续性保障**：配额耗尽不再导致 AI 功能突然中断，有 BYOK 的用户会议体验完全不受影响；
2. **费用透明度**：UI 在降级时明确告知费用来源，用户始终知道当前使用的是哪个 Key；
3. **自助解决路径**：无 BYOK 的用户可以在暂停界面当场配置 Key 立即恢复，不需要联系管理员后重新安排会议；
4. **授权边界清晰**：个人 Key 默认仅用于自己发起的会议，"为他人兜底"默认关闭，避免意外扣费纠纷；
5. **分层容错**：降级链覆盖了团队池→个人→其他成员的多层级兜底，大大降低会议中 AI 完全不可用的概率。

### 负面影响

1. **实现复杂度提升**：LLM 调用层需要支持动态 Key 切换和重试，预计增加 200-300 行核心代码；会议状态机增加 3 个新状态，状态转换路径增多；
2. **费用争议风险**：即使用户看到了提示，事后可能仍会抱怨"我不知道会扣我个人的钱"。需要通过二次确认、使用记录可查、首次降级时弹出更明显的通知来缓解；
3. **数据合规问题**：某些企业的合规策略禁止将工作数据发送到个人 API Key（可能绕过企业的数据保留策略）。需要提供组织级开关"禁止本团队成员使用个人 BYOK"；
4. **Key 切换期间的延迟**：检测到配额错误→切换 Key→重试调用会增加一次 LLM 调用的延迟（约 1-3 秒），实时功能（如实时字幕旁的 AI 标签）可能出现短暂卡顿；
5. **审计要求**：使用个人 BYOK 处理团队会议内容需要记录在审计日志中（谁的 Key、什么时间、处理了哪个会议），增加审计日志量。

### 缓解措施

- **首次降级强提示**：用户第一次触发个人 BYOK 降级时，弹出半模态对话框（需要点击确认），告知"后续 AI 调用将使用你的个人 Key 扣费"，并提供"本次会议不再使用我的 Key"选项；
- **组织级控制**：团队 Admin 可在团队设置中关闭"允许成员使用个人 API Key 兜底"，关闭后所有降级路径终止于 AI_PAUSED 状态（等同选项 A 行为）；
- **用量透明**：个人设置中提供"BYOK 使用记录"页面，展示每次使用个人 Key 的会议、时间、Token 消耗估算；
- **Key 切换缓存**：一旦在某个会议中降级到个人 Key，在会议剩余时间内持续使用该 Key（除非它也失效），避免反复切换带来的延迟和不稳定；
- **重试超时保护**：降级重试设置总超时上限（10 秒），超过则直接进入 AI_PAUSED 状态，不无限等待；
- **审计日志**：每次 Key 切换都写入审计日志（`audit_log` 表），记录 `event_type: 'llm_key_fallback'`，包含原 Key 来源、新 Key 来源、触发原因、会议 ID、触发用户 ID；
- **组织合规模式**：对于开启了"合规模式"的团队（如企业版客户），自动禁用个人 BYOK 降级，并在配额耗尽前 10% 就开始强预警，给管理员充足时间充值。

### 降级链路的状态持久化

会议的 Key 来源状态需要持久化到数据库（不仅是内存状态），以应对：

1. 会议进行中服务重启——重启后能恢复到降级后的 Key，而不是重新尝试已耗尽的团队 Key；
2. 多副本部署——WebSocket 连接可能被路由到不同副本，需要共享当前 Key 来源状态（通过 Redis 或数据库）；
3. 会后计费对账——纪要生成等会后异步任务使用的 Key 可能与会议中不同，需要记录清楚。

```sql
-- meetings 表扩展
ALTER TABLE meetings ADD COLUMN llm_key_source VARCHAR(30) NOT NULL DEFAULT 'team';
ALTER TABLE meetings ADD COLUMN llm_key_owner_user_id INTEGER REFERENCES users(id);
ALTER TABLE meetings ADD COLUMN llm_key_fallback_time TIMESTAMPTZ;
ALTER TABLE meetings ADD COLUMN llm_key_fallback_reason VARCHAR(50);
```
