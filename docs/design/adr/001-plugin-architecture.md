# ADR-001: 插件化架构作为核心扩展机制

| 字段 | 值 |
|------|-----|
| 编号 | ADR-001 |
| 状态 | Accepted |
| 日期 | 2026-07-19 |
| 影响范围 | 整个后端架构、核心模块边界、插件接口契约、部署模型 |

## 背景

Conclave 在 v0.1 至 v0.2 阶段采用单体架构实现，多租户隔离、团队管理、用量计费、第三方认证（OAuth/SAML）、审计日志、Webhook 通知等横切关注点与业务逻辑直接硬编码在核心代码库中。随着功能迭代出现以下问题：

1. **核心臃肿**：核心 `app/` 目录下已有 14 个直接依赖 LLM/Meeting 模块的业务子模块，单测覆盖率因耦合严重持续下降至 58%；
2. **无法独立替换**：计费模块需要从 Stripe 迁移到内部结算系统时，改动波及 11 个文件，跨越 `db/`、`api/`、`services/` 三层；
3. **无法按需启停**：私有化部署客户不需要团队管理功能，但代码仍然被打包并初始化相关数据库连接，增加了启动时间和故障面；
4. **团队协作冲突**：不同业务线同时修改核心模块造成大量合并冲突，功能分支合并周期平均 3.2 天；
5. **测试隔离差**：核心单测必须依赖完整的数据库、Redis、消息队列环境，无法对"纯会议生命周期"逻辑做快速单元测试。

我们需要一种可插拔的扩展机制，使核心保持精简稳定，业务能力和横切关注点通过独立模块装载，支持按需启用、独立测试、独立演进。

## 决策

采用**插件化单体（Plugin-based Modular Monolith）**架构作为 Conclave 后端的核心扩展机制。核心与插件的职责划分如下：

**核心（Core）仅保留以下最小职责：**

- 会议生命周期管理（创建/开始/结束/归档 Meeting）；
- LLM 调用抽象层（统一 Provider 接口、流式响应、重试/熔断）；
- 代码沙箱（Sandbox）执行环境管理；
- 插件运行时（PluginRegistry、Hook System、生命周期管理）；
- 事件总线（EventBus，进程内同步+异步事件分发）；
- 核心数据模型（User、Meeting、Transcript 基础表）；
- 基础 HTTP/WebSocket 框架与路由注册机制。

**所有其他能力一律通过插件实现：**

- 横切关注点：auth、billing、audit、ratelimit、webhook；
- 业务功能：team（团队管理）、workspace、template、analytics；
- 集成类：slack、feishu、zoom、notion。

**插件通过统一的 Hook System 与核心交互：**

- 核心在关键执行点定义标准钩子（Hook Point），例如 `before_meeting_start`、`resolve_llm_api_key`、`after_transcript_saved`、`on_billing_charge`；
- 插件实现对应钩子的 Mixin 接口，在初始化时向 PluginRegistry 注册；
- 钩子分为"拦截型（Interceptor）"与"观察型（Observer）"两类，调用语义见 ADR-004；
- 插件之间不直接依赖，如需通信通过核心 EventBus 发布/订阅事件。

插件以 Python 模块形式存在（`conclave/plugins/<name>/`），每个插件包含一个继承 `BasePlugin` 的入口类，声明自身元数据（名称、版本、层级、依赖的其他插件、所需钩子）。

## 选项对比

| 选项 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 选项1：单体硬编码，继续在核心内加 if/else 分支 | 实现最简单，无需额外抽象；调用链路直观 | 核心持续膨胀；横切逻辑污染业务代码；无法按需裁剪；测试成本指数上升；新人上手曲线陡峭 | 否决，已在 v0.2 中暴露出不可持续的维护问题 |
| 选项2：微服务拆分，每个业务域独立服务 | 物理隔离彻底；可独立部署扩缩容；团队边界清晰 | 引入分布式系统复杂度（网络延迟、分布式事务、服务发现）；Conclave 目前部署规模（单集群 <50 节点）不足以摊薄微服务成本；本地开发环境需要启动 8+ 服务；跨服务事务（如"创建会议-扣减配额-写审计"）实现复杂 | 否决，过度设计，与当前团队规模和部署形态不匹配 |
| 选项3：插件化单体（选定） | 核心精简；插件可独立开发/测试/启停；保留单体部署简单性（单进程、单数据库事务）；通过钩子解耦，边界清晰；未来可按需将重插件拆为独立服务 | 需要设计严谨的插件接口和钩子契约；钩子调用链调试复杂度略增；插件间版本兼容需要治理；需要配套插件加载/隔离/错误处理机制（见 ADR-003） | 选定，在保持单体部署优势的同时获得良好的模块化与可扩展性 |

## 后果

### 正面影响

1. **核心精简**：预计核心代码量从当前 18k 行缩减至 8k 行左右，核心单测覆盖率目标提升至 90%+；
2. **独立演进**：插件可独立发布版本，billing 插件从 Stripe 迁移只需替换插件内部实现，不影响核心；
3. **按需部署**：私有化客户可通过配置文件启用/禁用插件，最小部署仅需 core+auth 两个模块，启动时间从 12s 降至 4s；
4. **测试隔离**：插件可单独编写单元测试，使用核心提供的 MockHookBus 即可完成，无需依赖完整环境；
5. **团队边界**：不同业务团队认领不同插件，代码评审范围明确，合并冲突大幅减少；
6. **生态扩展**：未来可支持第三方插件（通过签名校验+沙箱限制），为 Conclave 构建合作伙伴生态预留接口。

### 负面影响

1. **接口设计成本**：钩子点的定义需要预先思考充分，后期修改钩子签名会破坏所有实现该钩子的插件，需要严格的版本化和废弃流程；
2. **调试链路变长**：一个请求可能穿过 5-8 个插件的钩子，需要完善的 trace_id 透传和钩子调用日志（在 core 中统一埋点）；
3. **插件间依赖治理**：某些插件可能依赖其他插件（如 analytics 依赖 team），需要在 PluginRegistry 中实现拓扑排序与依赖校验；
4. **性能开销**：钩子分发和事件总线有少量函数调用开销，经压测评估单请求增加 <2ms，在可接受范围内；
5. **学习曲线**：新开发者需要先理解 Hook System 和插件生命周期，而非直接阅读业务代码，需要配套文档和示例插件。

### 缓解措施

- 钩子接口在 `conclave/core/hooks/` 中以 TypeDict/Mixin 形式明确定义，变更需经过 RFC 评审；
- PluginRegistry 在 DEBUG 模式下输出完整的钩子调用链与耗时，便于排查问题；
- 插件依赖声明在 `plugin.yaml` 中，启动时做循环依赖检测和缺失依赖快速失败；
- 提供 `cookiecutter-plugin` 模板，一键生成符合规范的插件骨架。

### 插件生命周期约定

每个插件在其生命周期中经历以下标准阶段，由 PluginRegistry 统一驱动：

1. **DISCOVERED**：扫描到插件模块，读取 `plugin.yaml` 元数据，但尚未 import；
2. **LOADED**：Python 模块 import 成功，`BasePlugin` 子类实例化完成；
3. **INITIALIZING**：调用插件 `async setup()` 方法，建立数据库连接、初始化缓存、注册 HTTP 路由；
4. **READY**：`setup()` 成功完成，插件开始接收钩子调用；
5. **DEGRADED**（仅 CROSSCUTTING/OPTIONAL）：运行中出现可降级错误，部分功能不可用但未完全摘除；
6. **SHUTTING_DOWN**：收到进程退出信号，调用 `async teardown()` 释放资源、刷新 outbox；
7. **FAILED/STOPPED**：初始化失败或已完成关闭。

所有插件必须实现 `setup()` 和 `teardown()` 两个异步方法，即使是空实现。生命周期状态变更会通过 Observer 钩子 `on_plugin_state_change` 广播，允许其他插件感知并做出反应（例如 team 插件在 auth 插件 READY 之后再启动团队同步任务）。

### 插件目录结构约定

每个插件必须遵循统一的目录结构，以便自动化工具识别、加载、测试：

```
conclave/plugins/<name>/
├── __init__.py          # 导出 Plugin 类
├── plugin.yaml          # 元数据：名称、版本、tier、依赖、所需钩子
├── plugin.py            # BasePlugin 子类，钩子实现入口
├── models.py            # ORM 模型（如需自建表）
├── migrations/          # Alembic 迁移脚本（插件自建表时）
├── api.py               # FastAPI 路由（通过 register_routes 钩子注册）
├── services/            # 插件内部业务逻辑
└── tests/               # 插件单元测试，独立于核心测试
```
