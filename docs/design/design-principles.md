# Conclave 设计原则（固化条款）

> 本文档把散落在架构讨论中的判断固化成可引用的条款，作为评审与取舍的依据。
> 条款分两类：终态原则（长期成立）与 MVP 约束原则（限定第一阶段范围）。
> 终态架构背景见 [`ideal-design.md`](./ideal-design.md)，落地推导见 [`mvp-plan.md`](./mvp-plan.md)。

---

## 一、终态原则（长期成立）

### 原则 1：RAG 五原则

检索不是“文档切碎 → 全量 embedding → top-k”，而是结构化、可按需激活的知识系统。

| 序 | 原则 | 含义 |
|---|---|---|
| ① 保真切块 | chunk 带结构边界、语义边界、来源偏移，不是裸文本块 |
| ② 图而非链 | chunk 间关系建模为图（chunk graph），而非线性 prev/next |
| ③ 多维提炼 | 每 chunk 至少提炼 short_summary / claims / definitions / actionable_items |
| ④ 术语归一 | 独立术语归一层，解决同词异义、异词同义、中英混用 |
| ⑤ 分层惰性 | 解析、embedding、重排、上下文扩展均按需触发，成本不打满 |

### 原则 2：借调三问法

引入新专家角色前须回答三问，裁决结果为 `reject / defer / approve_temporary / approve_frozen_scope`。

1. 借调目标是什么（缺知识 / 视角 / 执行力 / 裁决力）
2. 借调是否必要（现有能力是否已近似，是否只是想更细）
3. 不借调的代价（结论失真 / 风险遗漏 / 证据不足 / 不可落地 / 仅质量略降）

依据：多 Agent 编排的 router / subagent / handoff 权衡，本质是在控制、成本、上下文隔离间平衡，不是角色越细越好。

### 原则 3：Agent 个体记录保留原则

发言记录是可监督的行为数据，不是聊天历史。保留有层次：

- **全量保留原始记录**，选择性提炼长期特征。
- 三层收敛：原始发言层（不可变）→ 行为特征层 → 稳定画像层；只有稳定画像层反哺系统初始化。
- 直接把全部发言当长期记忆，会导致噪声积累、错误固化、人格漂移、token 成本失控。
- 个体唯一性建立于 Role / Style / Evidence Preference / Risk Appetite 等决策偏置参数，而非戏剧化人设。

### 原则 4：事件广播 Runner 设计原则

WebSocket 是出口，不是核心。

- 应用层只产出领域事件，不直接广播。
- Event Runner 统一负责事件标准化、去重、排序、节流、fan-out、replay、trace 注入。
- 出口层多 sink 分发（WS / Audit / Metrics / Trace / Notification）。
- 首期内存实现，架构预留 MQ / Event Bus；协议（EventBus / BroadcastRunner）先抽象定清。

### 原则 5：个体即决策偏置参数

首期不要把“人格”做成戏剧化人设，要做成“决策偏置参数”。没有真实履历时堆设定容易伪装过度、表演化。

## 二、MVP 约束原则（限定第一阶段）

### 原则 6：MVP 判断三问

任何模块在进入 v1 前先过这三关：

1. **没这个能不能跑 MVP？** 能 → 推迟；不能 → 保留最简版。
2. **是否影响主闭环？** 主闭环 = 输入 → 多 Agent 协作 → 输出结果。不影响的不做复杂版。
3. **有没有可验证收益？** 不能在 1–2 周内验证收益 → 不做复杂版本。

### 原则 7：反架构沉迷

设计超过实现阶段是当前最大风险。处置策略：

- 终态愿景只作“概念候选池 + 架构灵感库”，不作当前实现清单。
- 高级抽象（chunk graph、术语系统、惰性管道、完整 event runner、人格演化）只做最薄接口，后续填充。
- 先跑通主闭环，再以插件方式逐项引入高级特性。

### 原则 8：演进式架构

不妥协愿景，但不被愿景绑架。路径：极简内核先验证全部核心链路 → 逐步把终态特性作为插件装上去。每一步都有可验证产物。

## 三、LLM 确定性与证据原则

### 原则 9：分阶段温度

创意应来自系统结构（角色差异、立场碰撞、证据驱动），而非采样随机性。temperature 不全局锁死，按阶段分级：

| 阶段 | temperature | 理由 |
|---|---|---|
| clarify / cross_team / evidence_check / arbitrate | 0 | 必须可复现，不能发散 |
| intra_team | 0.3 | 允许有限发散，角色差异已提供足够多样性 |
| produce | 0.1 | PRD 要稳定，允许微小表达差异 |

即使讨论阶段用了 0.3，后续阶段有结论锁定链 + 一致性自检兜底，不会让发散导致悖论。

### 原则 10：证据来源三级分级

agent 论点必须标注证据来源，让用户区分"有据可查"与"待验证建议"：

| 标记 | 含义 | 强度 |
|---|---|---|
| `[doc:section]` | 上传文档中的证据 | 强 |
| `[web:url]` | Web Search 检索的外部证据 | 中 |
| `[common_knowledge]` | 通用工程实践或行业常识 | 弱，需用户验证 |
| `[assumption]` | 基于当前信息的推理假设 | 最弱，需确认 |

无文档证据时，evidence_check 标注 `common_knowledge:none` 并降级置信度，而非虚构证据。

### 原则 11：感知层工具端口

工具调用的结果**不直接进结论链**，而是作为"证据"注入 evidence_check，和 RAG 证据一起走仲裁。工具的不可靠性被仲裁层过滤，不会污染最终决策。

首期只做结构化检索（Web Search API），浏览器自动化与桌面感知终态预留接口但不实现（见 [`ideal-design.md`](./ideal-design.md) §10）。

## 四、现阶段取舍速查

| 终态特性 | 第一阶段处置 |
|---|---|
| Chunk Graph | 暂缓，先 keyword + embedding + rerank |
| 术语归一系统 | 暂缓，先手动术语表 |
| 惰性 embedding 管道 | 暂缓，先简化 |
| 完整 Event Runner | 暂缓，先内存广播 + WS push |
| 三层记忆 / 画像演化 | 已实现三层记忆（PostgreSQL 持久化），画像自动更新阈值调参待后续迭代 |
| 借调自动化 | 暂缓，先人工触发 + 三问表单 |
| 多 sink 可观测管线 | 暂缓，先单一 WS 出口 |
| Docker 执行 | 视情况，见 [`architecture-review.md`](./architecture-review.md) 风险评估 |
| Web Search 工具 | 已实现四种模式：stub / tavily / playwright / remote，详见 `tools/__init__.py` |
| 浏览器自动化 / 桌面感知 | 终态预留接口，不实现 |

> 注：上述“暂缓”不等于“放弃”。接口边界画好，后续按 [`ideal-design.md`](./ideal-design.md) 装配。
