## 一、核心理念

### 1.1 从"固定流程的工具"到"可进化的自主系统"

Conclave 当前是"会议讨论 → 产出文档"的闭环工具，采用固定六阶段状态机。终态目标是"可进化的自主系统"——Agent 能自主决定流程、自我优化、多视角评估。

这是质变，不是量变。演进路径分三阶段（近期/中期/远期），不是一次迭代能完成的。

### 1.2 核心原则

- **动态优先**：流程动态生成，阶段根据实时必要性自由组装，不固定
- **人在回路**：系统可自主生成代码、优化、迭代，但合并需人工审批，不全自动
- **多视角评估**：优化后通过多系统、多 Agent、多模型评估，不同团队关注点不同
- **成本可见**：token 预算控制是必须的（方案二），实时价格展示是可选的（方案一），两者互补

## 二、动态流程架构

### 2.1 当前问题

当前六阶段（clarify → intra_team → cross_team → evidence_check → arbitrate → produce）是硬编码的 `STAGE_ORDER`，前端也是固定展示。问题：
- 简单任务不需要深度论证，跑完整六阶段是浪费
- 用户想实时介入时，只能通过 inject 追加消息，不能改流程
- 回退到前序阶段时，前端和存储的阶段没有动态改变或追加

### 2.2 目标形态

**阶段不再是预设的，而是根据对话内容动态构造的。**

- 议题复杂、涉及多方冲突 → 构造完整六阶段（含证据、仲裁）
- 议题明确、是开发任务 → 构造精简流程（分析→实现→验证）
- 议题是探索性的 → 开放式对话，用户随时插入

用户在任何流程中都能实时插话，系统根据插话决定：
- 是否回退到前序阶段
- 是否跳过后续阶段
- 是否插入新阶段

### 2.3 落地路径

**第一步：议题路由（近期）**
- clarify 阶段后，LLM 输出 `flow_plan`（该跑哪些阶段）
- 后续按 plan 走，而不是固定六阶段
- 六阶段变成"预设模板之一"，而不是唯一路径

**第二步：动态 DAG（中期）**
- 每个阶段是一个 Task
- 用 TaskGraph（阶段三已实现）的拓扑结构根据议题动态调整
- 常见组合顺序作为参考，但 Agent 不强制遵循

**第三步：前端动态展示（中期）**
- 前端阶段指示器不再固定写死四个/六个阶段
- 根据 flow_plan 动态渲染
- 回退或追加阶段时，前端和存储的阶段记录同步变化

## 三、Agent 自主执行循环

### 3.1 范式取舍

三种 Agent 循环范式对比：

| 范式 | 逻辑 | 适合场景 | 思考爆炸风险 |
|---|---|---|---|
| ReAct | 每步都重新推理"下一步做什么" | 动态未知环境 | 最高——反复调同一工具 |
| Plan-and-Execute | 先规划全局，再逐步执行 | 结构化任务分解 | 中——计划可能过时 |
| Re-Execution（Reflexion/Ralph Loop） | 外部循环控制，LLM 只做"修正" | 固定目标的迭代优化 | 最低 |

**取舍：Plan-Execute + 受控 Re-Execution**

不用纯 ReAct。Conclave 六阶段本身就是 Plan，把 produce 阶段改成 ReAct 等于推翻架构。应该用受控的 Re-Execution——LLM 在循环中只做"给定错误，如何修正"，不做"决策下一步"。

### 3.2 思考爆炸的正确定义

思考爆炸 ≠ 循环次数多。思考爆炸 = **话题偏离 + 重复讨论 + 突破边际**导致上下文被归档/缩略，LLM 遗忘原始目标开始发散。

防护手段（按重要性排序）：

1. **目标锚定**：每轮 prompt 带"任务是什么 + 已完成什么 + 当前这一步要做什么"三句话
2. **偏离检测**：每轮检测输出是否仍在任务边界内，偏离则拉回
3. **归档压缩而非丢弃**：上下文太长时生成结构化摘要（已完成步骤 + 关键决策 + 待办），而非粗暴截断
4. **循环次数硬上限**：max_rounds 作为物理兜底，不是主要防护

### 3.3 RefineLoop 设计

produce 阶段的 code_analysis/tested_system 加自迭代循环：

```
生成代码 → 沙箱执行 → 失败？
  ↓ 是
把 stderr 喂回 LLM，只让它做"修正"（每轮只带上一轮最终状态，不带完整历史）
  ↓
重新执行（≤5轮）
  ↓
成功或到上限 → 结束
```

关键约束：
- 每轮上下文 O(1)——只带"上一轮代码 + 报错"，不带更早历史
- LLM 只做修正，不做决策下一步
- max_rounds=5 硬上限
- 重复检测：连续两轮生成相同代码 → 判定卡死，终止

## 四、系统自我进化

### 4.1 目标

系统内置 Agent 应具备自主对当前框架进行优化和迭代的能力（自动化）。这是未来系统的必备能力，不增加这一步系统成长会受限。

### 4.2 落地形态：Meta-Agent + 人工审批

业界 self-improving agent 的做法是自动记录错误 → 识别模式 → 升级为永久规则 [1]。但 Conclave 要的更激进——Agent 把自己的代码库当作操作对象。

**核心流程**：
```
Meta-Agent 读取 metrics（测试覆盖率、执行耗时、错误率、证据召回率）
  → 识别改进点（"evidence_check 证据召回率低"）
  → 生成改进代码（修改 retriever.py）
  → 在沙箱跑测试验证
  → 生成合并请求或待办
  → 推送钉钉/微信通知给用户
  → 用户处理（说明操作原因）
  → 通过后合并到主分支
```

**关键约束**：不全自动合并。系统生成代码、编写功能、优化，但合并需人工审批。

### 4.3 多视角评估

优化后不能仅由系统内部评估。需要通过多系统、多 Agent、多模型进行评估。

原因：不同公司或大模型的创始团队、公司氛围和技术倾向不同，关注角度也不同。比如：
- OpenAI 系模型可能关注逻辑严谨性
- Anthropic 系模型可能关注安全性和可解释性
- 开源模型可能关注性能和资源效率

**评估流程**：
1. 系统内部评估（必有）：跑测试、检查覆盖率、验证功能
2. 多模型评估（可选）：把改进提交给不同厂商的模型，收集各自视角的评审意见
3. 多 Agent 评估（可选）：不同角色的 Agent 从不同维度评审（安全、性能、可维护性）
4. 人工最终决策

## 五、成本核算与预算控制

### 5.1 双轨制：方案二为主，方案一为辅

两者互补，不冲突。无论是否有方案一，方案二都一定能计算和展示。

**方案二（必须，token 计量）**：
- 不追求精确到分，做相对预算控制
- 每次会议设 token 上限（如 50 万）
- 超过 80% 警告，100% 停止
- 事后按厂商账单对账，不依赖系统内估算
- 当前 `llm_trace` 已记录每次调用的 input/output token，只需聚合 + 展示

**方案一（可选，实时价格）**：
- 维护 `pricing.json`，记录每家厂商每个模型单价
- 厂商有 pricing API 时定期拉取（如 OpenAI、Anthropic）
- 没有定价 API 的（促销活动），人工更新
- 实际成本 = input_tokens × 输入单价 + output_tokens × 输出单价
- 有定价数据时展示实时价格和实际消耗金额，没有时只展示 token

### 5.2 旁路调控

厂商调价不频繁时，pricing.json 做成旁路方式调控：
- 定价更新不阻塞主流程
- 有新定价时异步更新
- 无定价时回退到方案二（纯 token 展示）

## 六、沙箱开放策略

### 6.1 设计原则

沙箱默认开放网络，保留分级开关。用户创建会议时可选安全级别。

| 级别 | 网络 | 资源 | 适用场景 |
|---|---|---|---|
| L1（当前） | 断网 | 256MB，只读 | 简单脚本，预装库 |
| L2（新增） | 限网（pypi 白名单） | 512MB，可写 tmpfs | pip install 数据分析 |
| L3（新增） | 全联网 | 1GB，可写 | Web Search、API 对接、浏览器自动化 |

### 6.2 未来能力

沙箱开放后支持：
- Web Search（实时检索外部信息）
- 数据 API 对接（调用第三方 API 获取数据）
- 驱动浏览器自动化获取数据（如 Playwright/Selenium）
- pip install 任意包（限网白名单或全联网时）

## 七、实施优先级

### 近期（探索边界）
1. **议题路由**：clarify 后动态生成 flow_plan，打破固定六阶段
2. **RefineLoop**：produce 阶段代码自迭代修复
3. **token 预算仪表盘**：聚合 llm_trace，前端展示已消耗/剩余

### 中期（自我进化）
1. **Meta-Agent 改进建议生成**（不自动合并）+ 钉钉/微信通知
2. **偏离检测 + 归档压缩**：防止思考爆炸
3. **动态 DAG 流程**：阶段动态组装，前端动态展示
4. **多视角评估**：多模型/多 Agent 评审

### 远期（完全动态）
1. 完全动态流程 + 多 Agent 协作
2. 沙箱全开放（Web Search、浏览器自动化、API 对接）
3. MCP 服务器协议集成（知识库挂载 + 交叉映射）

## 八、关键约束与用户偏好

- **不全自动合并**：系统可自主生成代码、优化，但合并需人工审批
- **钉钉/微信通知**：改进建议生成后推送通知，用户处理
- **前端风格**：当前样式用户喜欢，不要随意改动
- **commit 规范**：必须完善、分组清晰
- **沟通记录沉淀**：所有讨论需归档到文档，方便后续回顾
